Testing and linting flow

A single-page map of how Piclaw validates changes locally and in CI — from focused developer loops, through lint/typecheck and staged tests, up to the canonical fast CI contract and the full local quality gate.

Why the flow looks like this
Piclaw is a Bun + TypeScript repo with runtime, web, vendored assets, and a large test surface. The validation flow is intentionally layered:
  • fast local iterations for touched files and focused regressions
  • repo-wide lint + typecheck for static safety
  • controlled staged tests to reduce memory spikes and SQLite concurrency issues
  • a canonical fast CI target that exercises the highest-signal checks cheaply
  • a full quality gate that adds pack/stale-dist/import-boundary/static-analysis checks

1. The flow at a glance

The repo uses a layered workflow rather than one giant command for every edit. This keeps normal work fast while still preserving a stronger final gate.

Focused developer check

Run only the tests that cover the touched area, usually from runtime/, with SQLite-safe sequential execution.

cd runtime && bun test --max-concurrency=1 …

Static validation

Run lint and typecheck via repo-aware wrapper scripts so tooling is resolved from the worktree and installed automatically if missing.

bun run lint
bun run typecheck

Default full test path

bun run test resolves to the controlled staged test runner, not a single monolithic bun test process.

bun run test

Fast CI contract

The canonical GitHub Actions path runs a deliberately bounded contract: selected check scripts, focused tests, and the full web build.

make ci-fast

Full local quality gate

Before bigger merges or release prep, run the superset quality command with lint, typecheck, tests, and structural hygiene checks.

bun run quality

The same concepts are exposed through both root-level Bun scripts and top-level Make targets. The Makefile is the human-friendly surface; package.json holds the actual script composition.

3. Linting and typecheck flow

Static validation is funneled through runtime/scripts/repo-dev-command.ts, which resolves repo-local binaries, auto-installs them if missing, and standardizes command invocation.
Lint path

ESLint via repo-dev wrapper

The lint command resolves the repo root, switches into runtime/, and runs:

eslint --config runtime/eslint.config.js src/**/*.ts test/**/*.ts
  • ignores dist/**, web/static/**, and node_modules/**
  • uses TypeScript parser/plugin plus JS recommended rules
  • project overrides include unused-vars enforcement and relaxed any/ts-comment rules
Typecheck path

Two related but distinct commands

The standard typecheck path is non-emitting:

tsc --noEmit -p runtime/tsconfig.json

There is also a build-oriented TypeScript path:

tsc -p runtime/tsconfig.json
  • typecheck = static safety, no build output
  • build = validation/inspection output under runtime/generated/dist
  • build clears old generated dist first

4. How the default test runner works

Piclaw does not treat “full test suite” as a single long Bun process by default. Instead it uses a controlled runner designed to keep the suite stable and observable.
Discovery

Find test files automatically

Discovers runtime/test/**/*.test.ts unless explicit files are passed.

Staging

Chunk into stages

Splits files into stages of 25 by default, running one stage after another.

Execution

Fresh Bun process per stage

Each stage runs in a fresh child process to bound heap/resource accumulation.

Concurrency

SQLite-safe by default

Forces --max-concurrency=1 unless the caller already supplied a different value.

Isolation

In-memory DB by default

Sets PICLAW_DB_IN_MEMORY=1 unless explicitly disabled.

Telemetry

Memory-aware reporting

Samples process-tree RSS and cgroup memory, emits a JSON report to runtime/generated/controlled-test-report.json.

Runner behavior Why it exists
Sequential stage execution Reduces flaky concurrency interactions and keeps SQLite access safe.
Fresh runtime per stage Prevents one huge Bun process from hiding stage-specific memory spikes.
Peak RSS + cgroup sampling Makes test-memory regressions measurable rather than anecdotal.
Passthrough args + explicit files Keeps the staged runner usable for focused suites, not just all-tests runs.

5. The fast CI contract

GitHub Actions intentionally runs a bounded contract instead of the entire local quality superset on every push/PR.
Workflow

.github/workflows/ci.yml

  • checkout
  • setup Bun
  • bun install --frozen-lockfile
  • make ci-fast

This keeps the CI contract legible and anchored to one repo-visible target.

What ci-fast actually does

High-signal fast checks

bun run check:silent-swallows bun run check:structured-logging cd runtime && PICLAW_DB_IN_MEMORY=1 bun test test/channels/web/tool-status-hints.test.ts test/web/status-hints.test.ts cd .. && make build-web
  • static guardrails for silent catch/logging regressions
  • two focused status-hints tests
  • full web build, which itself reruns web-build-related tests

6. The full local quality gate

The strongest repo-local gate is broader than CI fast. It is intended for higher-confidence local validation before larger merges, release prep, or more invasive refactors.
Phase Command What it catches
Static correctness bun run lint
bun run typecheck		 Style/bug patterns + TS structural breakage
Behavioral regressions bun run test Controlled staged runtime/web/test regressions
Packaging hygiene bun run check:pack-hygiene
bun run check:stale-dist		 Broken package contents and stale build artefacts
Architecture constraints bun run check:import-boundaries
bun run check:unused-exports		 Boundary violations and dead exported surface
Code health audits bun run check:hook-tdz
bun run check:silent-swallows
bun run check:structured-logging		 Runtime foot-guns and repo-wide hygiene policies

In practice: ci-fast is the canonical shared contract; quality is the stronger local “I want high confidence” gate.

7. Practical day-to-day flow

This is the most useful operational reading of the repo’s test/lint system: which commands to run when you are changing code.
Fastest loop

While actively editing

  • run focused tests in runtime/test/… for the touched surface
  • rerun bun run lint and bun run typecheck before considering the change stable
  • use raw or focused Bun test invocation when you only need a narrow regression pass
Before pushing

Canonical shared gate

  • run make ci-fast
  • this matches the GitHub Actions CI contract
  • if your change touches web bundles, this is especially important because build-web includes bundle validation tests
Before bigger merges

High-confidence local sweep

  • run bun run quality
  • optionally run make build-piclaw if you need the complete vendor + web + TS build path
  • for install/release work, add make local-install or smoke scripts after validation
When debugging test memory

Use the controlled runner report

  • inspect runtime/generated/controlled-test-report.json
  • adjust stage size / sampling if you need sharper memory evidence
  • use the report as an audit artifact, not just a pass/fail signal

8. Key takeaways

Mental model

There is no single “one true” command

Piclaw’s testing/linting flow is intentionally layered: narrow checks for editing speed, a bounded shared CI contract, and a broader local quality gate when you want stronger assurance.

Most important fact

Default tests are staged on purpose

The default bun run test path is not just cosmetic wrapping — it materially changes execution strategy to control memory growth and improve suite stability.

Fast CI scope

CI optimizes for signal, not exhaustiveness

make ci-fast deliberately checks a focused slice: important static audits, targeted tests, and a full web build. That keeps pull-request feedback quick and reliable.

Full gate

Quality is the local superset

If you need the strongest local confidence short of release smoke tests, bun run quality is the repo’s best single umbrella command.
